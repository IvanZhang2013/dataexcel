1.类的设计一团糟
本来遵循的原则是  抽象类来定义属性  接口来定义行为 
 
构造对象的话要集成抽象类实现接口  感觉会有点乱导致如果继承和接口继承少的话无法作用   感觉好像少一层    抽象类 
第二层的抽象类是不是用来继承第一个抽象类然后实现接口 
  
任务和操作划分得不够明晰导致无法看字明义
2。初始目标
设计的过程中变更太大造成的原因前期没有明确最后的设计的样子  导致在设计的过程中要频繁的更改 

感觉设计的思路错误  不应该是由小到大  这样的话会越来越小  应该是尽可能的大然后向小的落脚  
在变更的过程中逐渐丰富自己的内容  所以可以留必要的接口

3.文档 
设计的时候最好是有设计文档要不做完之后会发现  四不像，内容结构糟糕造成代码维护的麻烦


4.关于测试感觉需要使用junit 要不在后期类太多必须使用流程测试但是其实有些时候是不会有流程的  造成出错
